# 入口：写标书 (grant)

把选题写成中文基金申请书初稿：填题名/想法/背景 → AI 生成方案骨架 + 大纲（可确认/编辑）→ 分节撰写 → 引用核验。
代码：`frontend/src/modules/GrantModule.tsx`；样式：`frontend/src/styles.css`。

---

## 第 1 圈 · 2026-07-02

### 起因（批判要点）
派 agent 从「非 IT 医学/药学 PI 第一次写标书」视角批判，12 条，按严重度排序。摘录：

**高**
1. 主按钮被禁用（题名/报告都空）时没有任何解释，用户以为程序卡了/坏了，不知道「要先填题名」。
2. 全程没有持久的「我在第几步」进度指示；进入 writing/done 后确认面板整块隐藏，用户失去流程地图。
3. 题名标「必填」，但放行条件是 `title || report`（只填报告、题名留空也能开始）——规则与提示自相矛盾。

**中高/中**
4. 「① 生成大纲」与「一步到位直接写完」并排、主次不清，新用户不知点哪个、不知代价差别。
5. 顶部说明是一大段密集文字（6 个 strong + code），劝退式信息密度。
6. 导出只有 Markdown/Word，缺「复制全文」；编辑藏在 TipTap 里。
7. 确认面板的「收起」按钮名不副实：实际是 `setPhase("idle")` 把整个确认区重置消失（数据其实还在），用户误以为白填了/内容丢了。
8. 大纲行的「篇幅/budget」是无列头、无占位的裸输入框，用户不知填什么。
9. 「撰写前重新检索」复选框用 type-chip 样式且勾选后无选中态，重要开关被做得不起眼。
10. 长时间流式生成只有一行 status，无「正在写第 3/6 节」这类分节进度。
11. 「清空」是不可撤销的破坏性操作却无二次确认，紧挨生成按钮，误点全没。
12. 结果面板在 idle 时就常驻显示「等待开始」空框，制造空态噪音。

### 本轮取舍
三条最高severity（#1/#2/#3）都围绕「输入门槛」这一处，收敛成一个紧凑、低风险的修复；同时顺手修 #7 这个措辞误导。#2 完整 stepper、#4 主次、#5 精简说明、#6 复制、#11 二次确认等留待后续轮次。

### 改动
- `GrantModule.tsx:284`（#3）— 去掉题名字段上误导的红色「必填」标记：题名并非单独必填（题名或报告其一即可），保留「必填」会与实际放行规则矛盾。
- `GrantModule.tsx` form-actions 下方（#1+#3）— 新增**门槛提示**：当 `!hasInput` 时显示一行 muted 文案「开始前请至少填写项目题名，或粘贴一份选题调研报告——两者填其一即可生成」。一句话同时解释「按钮为什么点不动」和「到底要填什么」。
- `GrantModule.tsx:494`（#7）— 「收起」→「← 返回修改（内容不丢）」，并加 title 提示「返回上方表单继续修改；已填内容与已生成的大纲都会保留」，措辞与实际行为一致，消除「点了内容会丢」的误解。

### 预期效果
- 第一次用的 PI 在题名/报告都空时，能立刻看到该填什么，不再对着灰按钮以为程序坏了。
- 「必填」不再与实际规则打架，提示自洽。
- 「返回修改」按钮不再吓退用户，明确内容不会丢。
- `npm run build` 通过（tsc + vite，无报错）。

### 未做 / 留待后续轮次（grant 第 2 圈优先）
- **#2**：加一个持久的 stepper/步骤条（骨架→大纲→分节→核验），writing/done 阶段也能看到自己在第几步、能否回退。
- **#4**：厘清两条主路径的主次与代价差异（推荐先确认 vs 一步到位）。
- **#6**：结果区加「复制全文」按钮（与 idea 模块一致）。
- **#11**：「清空」加二次确认，避免误点丢全部资料。
- #5 顶部说明分步化/示例化；#8 budget 列头与占位；#9 重要开关视觉强调；#10 分节进度；#12 idle 空态收敛。

## 第 2 圈 · 2026-07-02

### 起因（批判要点）
派 agent 二轮评审，重点找功能性 bug 并核实第 1 轮遗留。摘录：

**严重·功能 bug**
- **genPlan 完全无错误处理**：`planGrant` 对任何 fetch/HTTP/JSON 失败都吞掉并返回空大纲（`sse.ts` fallback）。于是：①「按意见调整大纲」失败 → `setOutline([])` 把用户改了半天的大纲清空、「② 确认大纲并撰写」因 `outline.length===0` 变灰、无报错；②「生成大纲」失败 → 无条件 `setPhase("planned")` 弹出一个只有题名、其余全空、大纲为空的确认面板，用户误以为「AI 只生成了个标题」。
- **【第 1 轮改动引入的可用性回归】「返回修改（内容不丢）」是死路**：点它 `setPhase("idle")`，而确认面板只在 `phase==="planned"` 渲染 → 回到 idle 后**没有任何入口重新看到/使用**已生成的方案骨架与大纲；唯一前进路径「① 生成大纲」会全量重置、「一步到位」根本不带 scheme/outline。承诺「内容不丢」在体验上误导。

**核实第 1 轮遗留仍在**
- #11「清空」无二次确认，一键抹掉已完成整份初稿（原样仍在）。
- #6 结果区无「复制全文」（原样仍在）。
- #8 大纲行 budget 裸输入框无 label/placeholder（仍在）。
- #6(SSE) 若服务端未发 done/error 帧，phase 停在 writing，「逐节修改」永不出现、不入历史（健壮性）。

### 本轮取舍
优先修真实 bug 与自己上轮引入的回归：**genPlan 错误处理 + 「返回修改」死路**；再补两个第 1 轮遗留的高频/破坏性项：**清空二次确认、复制全文**。stepper(#2)、budget 占位(#8)、SSE 兜底(#6-sse)、顶部双写入口(#5) 留待后续。

### 改动
- `GrantModule.tsx`（genPlan）— 套 `try/catch/finally`；**planGrant 返回空大纲即视为失败**：`setError(...)` 并 `return`，不清空已有大纲、不进空的确认面板；`finally` 复位 planning/status。修掉「改大纲失败被清空」「生成失败弹空面板」两个静默 bug。
- `GrantModule.tsx`（#4 回归）— idle 阶段且已有 scheme+outline 时，门槛提示下方加「继续上次生成的大纲 →」按钮（`setPhase("planned")`），让「返回修改（内容不丢）」名副其实——回到 idle 后仍能取回已生成的大纲。
- `GrantModule.tsx`（#11）— `reset()` 在有输入或已生成内容（title/report/idea/scheme/sections）时先 `confirm("将清空全部输入与已生成的方案/初稿，且不可撤销。确定？")`，避免误点丢全部。
- `GrantModule.tsx`（#6）— 结果工具栏加「复制全文」按钮（带「已复制 ✓」），与 idea/analyze/imrad 一致。

### 预期效果
- 生成/调整大纲失败不再静默清空或弹空面板，而是明确报错并保留现有内容，可重试。
- 「返回修改」不再是死路，用户能回到并继续已生成的大纲，「内容不丢」兑现。
- 误点「清空」有拦截，保住辛苦写的申请书。
- 申请书全文可一键复制去粘进单位模板/填报系统。
- `npm run build` 通过。

### 未做 / 留待后续轮次（grant 第 3 圈优先）
- **#6-sse（健壮性）**：`streamGrant` 若无 done/error 帧，`startWrite` 用 finally 兜底 `setRunning(false)` 且把 `writing→done`，避免「有正文却无逐节修改/不入历史」的错乱态。
- **#2** 持久步骤条；**#5** planned/writing 阶段折叠顶部双写入口，避免误点覆盖；**#8** budget 加列头/占位；**#10** 分节进度（第 X/N 节）。
