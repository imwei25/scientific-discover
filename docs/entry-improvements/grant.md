# 入口：写标书 (grant)

把选题写成中文基金申请书初稿：填题名/想法/背景 → AI 生成方案骨架 + 大纲（可确认/编辑）→ 分节撰写 → 引用核验。
代码：`frontend/src/modules/GrantModule.tsx`；样式：`frontend/src/styles.css`。

---

## 第 1 圈 · 2026-07-02

### 起因（批判要点）
派 agent 从「非 IT 医学/药学 PI 第一次写标书」视角批判，12 条，按严重度排序。摘录：

**高**
1. 主按钮被禁用（题名/报告都空）时没有任何解释，用户以为程序卡了/坏了，不知道「要先填题名」。
2. 全程没有持久的「我在第几步」进度指示；进入 writing/done 后确认面板整块隐藏，用户失去流程地图。
3. 题名标「必填」，但放行条件是 `title || report`（只填报告、题名留空也能开始）——规则与提示自相矛盾。

**中高/中**
4. 「① 生成大纲」与「一步到位直接写完」并排、主次不清，新用户不知点哪个、不知代价差别。
5. 顶部说明是一大段密集文字（6 个 strong + code），劝退式信息密度。
6. 导出只有 Markdown/Word，缺「复制全文」；编辑藏在 TipTap 里。
7. 确认面板的「收起」按钮名不副实：实际是 `setPhase("idle")` 把整个确认区重置消失（数据其实还在），用户误以为白填了/内容丢了。
8. 大纲行的「篇幅/budget」是无列头、无占位的裸输入框，用户不知填什么。
9. 「撰写前重新检索」复选框用 type-chip 样式且勾选后无选中态，重要开关被做得不起眼。
10. 长时间流式生成只有一行 status，无「正在写第 3/6 节」这类分节进度。
11. 「清空」是不可撤销的破坏性操作却无二次确认，紧挨生成按钮，误点全没。
12. 结果面板在 idle 时就常驻显示「等待开始」空框，制造空态噪音。

### 本轮取舍
三条最高severity（#1/#2/#3）都围绕「输入门槛」这一处，收敛成一个紧凑、低风险的修复；同时顺手修 #7 这个措辞误导。#2 完整 stepper、#4 主次、#5 精简说明、#6 复制、#11 二次确认等留待后续轮次。

### 改动
- `GrantModule.tsx:284`（#3）— 去掉题名字段上误导的红色「必填」标记：题名并非单独必填（题名或报告其一即可），保留「必填」会与实际放行规则矛盾。
- `GrantModule.tsx` form-actions 下方（#1+#3）— 新增**门槛提示**：当 `!hasInput` 时显示一行 muted 文案「开始前请至少填写项目题名，或粘贴一份选题调研报告——两者填其一即可生成」。一句话同时解释「按钮为什么点不动」和「到底要填什么」。
- `GrantModule.tsx:494`（#7）— 「收起」→「← 返回修改（内容不丢）」，并加 title 提示「返回上方表单继续修改；已填内容与已生成的大纲都会保留」，措辞与实际行为一致，消除「点了内容会丢」的误解。

### 预期效果
- 第一次用的 PI 在题名/报告都空时，能立刻看到该填什么，不再对着灰按钮以为程序坏了。
- 「必填」不再与实际规则打架，提示自洽。
- 「返回修改」按钮不再吓退用户，明确内容不会丢。
- `npm run build` 通过（tsc + vite，无报错）。

### 未做 / 留待后续轮次（grant 第 2 圈优先）
