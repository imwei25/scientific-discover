# 入口：伦理材料 (ethics)

帮用户准备伦理送审材料（知情同意书 / 研究方案 / CRF / 数据使用承诺）。单选 tab、逐份填空 →
后端 `/api/ethics/render` 填充 .docx 模板下载。代码：`frontend/src/modules/EthicsModule.tsx`；样式：`frontend/src/styles.css`。

---

## 第 1 圈 · 2026-07-02

### 起因（批判要点）
派 agent 从「非 IT 医学/药学研究者第一次准备伦理送审」视角批判。两个结构性事实先框定：
①本模块**没有 AI 起草**（按钮写「生成中…」但只是把字段 POST 给后端填模板），用户要自己写每一条；
②是**单选 tab、一次一份**，无「我需要哪几份」的引导、无「打包下载全套」。据此 10 条，摘录：

**高**
1. **无必填校验——可以下载一份近乎空白的正式文件**：`download()` 对 `fields` 零校验，`FieldDef` 无 `required`，没有任何字段显示必填标记；讽刺的是 CSS 里有现成却从未被用的红星样式 `.field-label em`。首次送审者不知道伦理委员会把哪些项视为必交，工具会照样产出缺风险/隐私/联系方式的知情同意书 → 送审即被退回（正是本产品该防止的失败）。
2. 「生成」暗示会帮起草，但用户其实要独自写完每个硬骨头章节（样本量/统计方法等），16 个空框零脚手架。

**中**
3. 无复制、无可编辑输出，唯一出口是下载 .docx（很多伦理系统要求把文本粘进它自己的模板/网页表单）。
4. 「清空字段」是不可撤销的 native `confirm()` 抹除、无 undo。
5. **「从实验规划导入」在没有可导入内容时静默失败**：`doImport` 只填空字段，用户没用过实验规划时点了毫无反应、也无「未找到可导入内容」提示。
6. 报错是原始 HTTP/后端行话、无重试路径。
7. 研究方案是 16 字段一堵平墙，无分组/步骤/层级。
8. 进度「已填 X / Y 项」会把样板默认值也算进去、且不区分必填与选填，误导「离能送审还有多远」。

**低**
9. 预览用开发者语言（「后端模板」）、且经 CanvasSlot портал 到右画布，位置/可见性不可预测。
10. h1 / active-desc / tabs 三处重复同样信息；下载提示长句挤在按钮行里，窄屏换行破坏对齐。

### 本轮取舍
以 **#1（最高、最贴产品使命）** 为核心，顺带修 **#5（静默失败）** 这个一眼可见的坏体验。AI 起草(#2)、复制/可编辑(#3)、清空 undo(#4)、字段分组(#7)、报错友好化(#6) 等留待后续轮次。

### 改动
- `EthicsModule.tsx`（#1）— 新增 `REQUIRED: Record<TemplateId, string[]>`：为每种材料标出「缺失最易被退回」的核心必填项（如知情同意书：研究名称/研究目的/风险/隐私保护/研究者/联系方式/机构/日期）。保守只标不可争议的核心项，避免过度拦截。
- `EthicsModule.tsx`（#1）— 必填字段标签渲染红色「必填」标记（复用现成但一直没被用的 `.field-label em` 样式）；`missingRequired` 非空时**禁用下载按钮**，并在下方红字列出「还有 N 个必填项未填，补齐后才能下载：X、Y…」。与 grant/plan 的「禁用即解释」一致。
- `EthicsModule.tsx`（#5）— `doImport` 现在比较导入前后已填项数：导入到内容显示「已从『实验规划』导入 N 项」，否则显示「未找到可导入内容——请先在『实验规划』里填写或生成方案，再回来导入」；用中性 `msg` 状态（区别于红色 `err`），5s 自动消失。

### 预期效果
- 用户一眼看清哪些项是伦理委员会必交的，且不补齐无法下载 → 从源头减少「送审被退回」。
- 「从实验规划导入」不再点了没反应，明确告知导入了几项或为何没有。
- `npm run build` 通过（tsc + vite，无报错）。

### 未做 / 留待后续轮次（ethics 第 2 圈优先）
- **#2**：给硬骨头字段（样本量/统计方法/研究设计）加「查看示例」或从 plan/idea 带入的起草脚手架。
- **#3**：预览区加「复制全文」按钮（与 idea 一致）。
- **#7**：研究方案 16 字段按「背景 / 设计 / 统计 / 署名」分组 + 渐进展开。
- **#4** 清空加 undo/软删除；**#6** 报错友好化 + 重试；**#8** 进度只数必填、区分必填/选填；**#9/#10** 预览措辞与信息层级去重。

## 第 2 圈 · 2026-07-02

### 起因（批判要点，聚焦 bug + 核实第 1 轮遗留）
派 agent 二轮评审，实测确认两个真实 bug（且都削弱了第 1 轮的改动）：

**bug**
1. **下载失败时后端真实错误被前端静默吃掉**：前端读 `(await resp.json()).detail`（`EthicsModule.tsx`），但后端 `main.py:924,926` 的错误分支返回的是 **text/plain**（正文即错误信息，非 JSON `{detail}`）。于是 `resp.json()` 必抛 → 被 `catch{}` 吞 → detail 恒空 → 用户永远只看到「服务返回 500」，而后端明明回了「生成失败: 某字段…」这类可自救信息。第 1 轮遗留 #6（报错技术化）因此依旧成立。
2. **`importFromPlan` 无条件塞样板默认 → 「已导入 N 项」虚报**：`自愿原则/隐私保护/日期` 三项无条件 `fill`，`doImport` 用「导入前后已填数之差」判断，对知情同意书即使 plan 完全为空也必然 `added≥3` → 永远走成功分支，**第 1 轮新加的「未找到可导入内容」反馈对该材料永不触发**。用户 plan 里啥都没写却看到「已导入 3 项」，误以为规划被带过来了。

**核实第 1 轮遗留仍在**：#3 预览无复制、#7 研究方案 16 字段无分组、#8 进度把选填/样板都算进分母且与必填下载门不一致、#9 预览抬头开发者措辞、#10 h1 与 tab 信息重复。
（澄清：下载失败**不会锁死**按钮——`finally{setBusy(false)}` 齐全；切 tab 也不串味——各材料独立 storageKey。真隐患是「下载 in-flight 时切 tab 丢反馈」。）

### 本轮取舍
按 cycle 2「先修 bug」原则，修 #1（错误被吞）+ #2（导入虚报，且它破坏了第 1 轮的诚实反馈）；再补一个高频一致性件 #3/#9（预览复制 + 抬头改用户语言）。分组(#7)、进度口径(#8)、清空 undo(#4)、in-flight 切 tab(#6-new) 留待后续。

### 改动
- `EthicsModule.tsx` `download`（#1）— 错误正文改用 `await resp.text()`（后端就是 text/plain），把「生成失败: 某字段…」这类可自救信息如实透出，而非永远「服务返回 500」。
- `EthicsModule.tsx` `importFromPlan`（#2）— 改为返回 `{ fields, planFilled }`：只统计**真正来自 plan 内容**的填充；三项通用样板默认（自愿/隐私/日期）**仅当确有 plan 内容导入时才补**、且不计入 planFilled。`doImport` 用 planFilled 判断——plan 为空时如实提示「未找到可导入内容」，不再虚报。（顺带删掉不再使用的 `countFilled`。）
- `EthicsModule.tsx` 预览（#3/#9）— 预览抬头加「复制全文」按钮（带「已复制 ✓」）；抬头文案从「Word 版样式由后端模板决定」改为用户语言「以下为填写内容速览，正式排版以下载的 Word 为准」。

### 预期效果
- 下载失败时用户能看到真实原因、据此自救（如改掉某字段的非法字符），而不是对着「500」反复点。
- 「从实验规划导入」的反馈恢复诚实：plan 为空就说「未找到」，不再虚报「已导入 3 项」。
- 填写内容可一键复制去贴进机构自有 Word 模板；预览抬头不再是开发者黑话。
- `npm run build` 通过。

### 未做 / 留待后续轮次（ethics 第 3 圈优先）
- **#7** 研究方案 16 字段按「背景/设计/统计/署名」分组（给 FieldDef 加可选 group）。
- **#8** 进度拆成「必填 a/b · 选填 c/d」，与下载门同一套 requiredKeys。
- **#6-new** 下载 in-flight 时禁用 tab 切换，避免丢「生成中/失败」反馈；**#4** 清空改应用内确认条 + 「撤销」。
