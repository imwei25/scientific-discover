# 入口：数据分析与写作 (analyze)

上传数据(csv/excel) → 本地统计分析 + 出图(PNG/SVG/PDF) → AI 提炼核心观点、写论文段落
（数字由本地 Python 算，不让 LLM 算数）。含 general / 森林图 / KM / ROC 多种分析。
代码：`frontend/src/modules/AnalyzeModule.tsx`；样式：`frontend/src/styles.css`。

> 注：本模块近期刚被重点强化过「成功率」（探索轮/方法路由/修复循环），本轮聚焦界面/交互/可用性层面。

---

## 第 1 圈 · 2026-07-02

### 起因（批判要点）
派 agent 从「非 IT 医学/药学研究者第一次分析自己的数据」视角批判，12 条，按严重度排序。摘录：

**高**
1. 上传后**完全没有数据预览**，用户无法确认文件读对了（合并单元格/多余表头/乱码），只能盲点「开始分析」，信任崩塌。
2. **xlsx + KM/ROC 是死路**：`ColMapper` 在无表头时提示「请直接输入列名」，但组件里只有 `<select>`、**根本没有输入框**，也没有运行按钮——功能彻底走不通，而 xlsx 是医学最常见格式。
3. 「真实计算结果」只有折叠的原始 stdout（等宽 `<pre>`），没有可读的统计结果表，把最有信服力的证据藏了起来。

**中**
4. 「分析类型」下拉切换整块改变界面，且森林图根本不用上传的文件（`runForest` 只用手填表），心智模型跳变。
5. 结果工具栏按钮被 `.result-toolbar` 的 `space-between` 拉散（3 个按钮直接平铺，未用 `.result-actions` 包裹），空隙大而不均、观感松散。
6. 结论区没有「一键复制」，医学用户最想要的「粘进 Word/发给导师」动作缺失。
7. 刷新后残留旧结果但 `file` 不持久化 → 显示完整结论却因 `disabled={!file}` 无法重跑，错乱。
8. 长耗时分析只有一行状态文字，无阶段/进度/耗时。
9. 报错一行红字 + 原始技术信息、无重试。

**低**
10. forest/km/roc 结果无免责声明、统计量(I²/Log-rank p/AUC…)零解释。
11. 无示例数据/模板/默认问题，冷启动门槛高。
12. 拖拽区太单薄（细 bar），不像可拖放区域。

### 本轮取舍
以 **#2（xlsx 死路，最高优先的可走通性 bug）** 为核心，顺带两个低风险打磨：#5 工具栏编组、#6 复制结论（与 idea 一致）。数据预览(#1)、可读统计表(#3)、分阶段进度(#8) 等较大改动留待后续轮次。

### 改动
- `AnalyzeModule.tsx` `ColMapper`（#2）— 无表头（xlsx 等）时不再是死路：把每个列映射从「只有 `<select>`」退化为**可手动输入列名的 `<input>`**，并保留运行按钮 + 一句「需与首行列名完全一致」的提示。因 `ColField` 的 value/onChange 与 select 路径共用同一状态，KM/ROC 运行逻辑无需改动，零后端/契约变更。
- `AnalyzeModule.tsx` 结论工具栏（#5）— 把「用此结论去排版 / 导出HTML / 导出MD」三个按钮用 `<div className="result-actions">` 包裹，右对齐编组，不再被 `space-between` 拉散。
- `AnalyzeModule.tsx` 结论工具栏（#6）— 新增「复制结论」按钮：一键复制结论到剪贴板，短暂显示「已复制 ✓」/失败态；用局部 `copyState` 状态，不触碰会带 spinner 的 `status`。

### 预期效果
- xlsx 用户也能做 KM/ROC 分析（手动输入列名），补上一条彻底走不通的路径。
- 结论工具栏按钮编组紧凑、观感精致。
- 用户可一键复制结论去粘贴，省掉下载文件或手动划选。
- `npm run build` 通过（tsc + vite，无报错）。

### 未做 / 留待后续轮次（analyze 第 2 圈优先）
- **#1（高）**：上传后加数据预览（行列数 + 前几行 + 字段类型），让用户确认文件读对了。
- **#3（高）**：把原始 stdout 解析/呈现为可读的统计结果表（这是「可信」卖点的关键证据）。
- **#7**：`file` 缺失但有旧结果时给「结果对应的数据已丢失，请重新上传」提示，或清理陈旧结果。
- **#8** 分阶段进度（探索→生成→执行→修复）；**#9** 报错友好化 + 重试；**#4** 森林图不吃上传文件的跳变；**#10** 统计量 tooltip + 全路径免责；**#11** 示例数据；**#12** 更醒目的拖放区。

## 第 2 圈 · 2026-07-02

### 起因（批判要点，聚焦状态/健壮性 bug）
派 agent 二轮评审，实测确认多个真实 bug：

**严重**
1. **中止(stop)后半截结论被当「已完成」写入历史**：`stop()` 只 `abort()`+`setRunning(false)`+清 status，不设 error、不清 conclusion；而历史保存 effect 条件是 `!running && !error && conclusion`，中止后三者恰好满足 → 把残缺分析 `addHistory`。用户主动中止（可能因发现分组错了），半截、可能自相矛盾的结论却进了历史、日后被误当定论。
3. **切分析类型 / 换文件时旧结果不清 → 跨文件串味**：`setChartType`/`pickFile` 都不清结果，渲染仅按 chartType 门控（隐藏而非销毁）。复现：文件 A 出通用结论 → 切 ROC → 换文件 B 跑 → 切回通用，屏幕仍是**文件 A 的结论**、顶部却是文件 B。数字与当前文件脱钩，是数据工具红线。
7-#1. 刷新/切项目后「有结论、无文件、开始分析变灰」的死态仍在（`file` 用 useState 不持久化，结果全 usePersistentState）。

**中·新**
5. **生成图注静默失败**：`genCaptions` 只有 try/finally、**无 catch**、不查 `resp.ok`；网络失败 → 未处理 rejection，用户毫无反馈、以为点了没反应。

**核实第 1 轮遗留仍在**：#3 无可读统计表、#6 上传无数据预览、#10 forest/km/roc 无免责/统计量无解释、#8(部分) 无步骤勾选。#11 无示例数据、#8(forest) `Number()||0` 静默把漏填当 0。

### 本轮取舍
按 cycle 2「先修 bug」原则，一次修掉 4 个状态/健壮性问题：**#2（脏数据入历史）+ #3（跨文件串味）+ #1（刷新死态提示）+ #5（图注静默失败）**。可读统计表(#3)、数据预览(#6)、全路径免责(#10)等较大的展示改造留待后续。

### 改动
- `AnalyzeModule.tsx`（#2）— 新增 `abortedRef`：`run()` 开始置 false、`stop()` 置 true；历史保存 effect 加 `&& !abortedRef.current`，手动中止的半截结论不再入历史。
- `AnalyzeModule.tsx`（#3）— 抽出 `clearResults()`（清 plan/code/charts/captions/output/conclusion/forest·km·roc 结果与错误 + 重置 savedRef）；`reset()` 复用它；**`pickFile` 换文件时调用它**，保证屏幕结论始终对应当前文件，消除串味。
- `AnalyzeModule.tsx`（#1）— 通用模式下 `!file` 但有持久化结果时，按钮下方提示「下方结果是上次分析的本地存档；原数据文件已不在，如需重新分析请重新上传文件」，把莫名变灰的按钮解释清楚。
- `AnalyzeModule.tsx`（#5）— `genCaptions` 加 `catch` 和 `resp.ok`/`d.ok` 分支，失败经 `reportLLMError` 给出可读提示，不再静默。

### 预期效果
- 用户主动停止分析后，残缺结论不会污染历史记录。
- 换文件后再也不会看到「结论来自旧文件、顶栏却是新文件」的错乱数字。
- 刷新后看到存档结果、按钮变灰时，知道是「原文件丢了，重新上传即可」而非「系统坏了」。
- 生成图注失败有明确反馈。
- `npm run build` 通过。

### 未做 / 留待后续轮次（analyze 第 3 圈优先）
- **#6 数据预览**（行×列 + 前 5 行，表头已解析可直接用）；**#3 可读统计表**（解析 stdout 出 p 值/效应量/n）。
- **#10** 把 `.analyze-disclaimer` 抽成共享组件挂到 forest/km/roc，统计量加 tooltip；**#8(forest)** 提交前校验 N>0、0≤event≤N，漏填标红而非计 0；**#11** 各类型加「载入示例数据」；**#9** 报错加「重试」按钮。
