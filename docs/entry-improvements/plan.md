# 入口：实验规划 (plan)

把研究想法变成可执行的实验方案 + 时间表，含交互式「样本量计算器」、SAP/DMP/知情同意生成、随机化分组表。
代码：`frontend/src/modules/PlanModule.tsx`；样式：`frontend/src/styles.css`。

---

## 第 1 圈 · 2026-07-02

### 起因（批判要点）
派 agent 从「非 IT 医学/药学研究者第一次做实验设计」视角批判，12 条，按严重度排序。摘录：

**高**
1. **「使用此参数」是假承诺——算出的样本量根本没进 AI 方案**：`useThisN` 把 N 存到 `plan:sampleSize` 并显示「已保存到方案」，但 `submit()` 生成方案时只传 `{idea, field, resources}`（L49-52），从不读 `ssChosen`。用户以为样本量已进方案，实际方案与之无关、可能自相矛盾。
2. 计算器藏在结果下方，与 AI 方案在空间/逻辑上割裂，工作流先后倒置（先生成、往下滚才发现计算器）。
3. 双比例基线率 p₁ 硬编码 0.3 不可改（L133/L196），真实基线率千差万别，锁死会算出错误 N。

**中**
4. 一行 5 个生成按钮（含 SAP/DMP 缩写）无换行、主次不分、点击前无解释。
5. 必填校验只有「置灰」，无文字提示。
6. 表头承诺「时间表/里程碑」，但产出只是纯 Markdown，无结构化时间表 UI。
7. 样本量结果缺失临床必备的失访率上浮/分配比例/置信区间。
8. 默认大字展示的 N 是前端 6 点查表+插值的近似值，误差要点按钮才暴露。

**低**
9. 验证结论塞进小灰字+多层括号，关键「是否可信」最不显眼。
10. 两个折叠器状态不一致，展开的计算器占大量纵向空间。
11. 效应量单滑块跨两场景语义不同、切换不重置，残留值可致计算失真。
12. 报错技术化、无重试入口。

### 本轮取舍
以 **#1（真实功能 bug）** 为核心——这是「交互式样本量」这个卖点实际不可用的根因；顺带做两个与前几轮一致的低风险收尾（#5 门槛提示、#4a 按钮换行）。#2 顺序重排、#3 p₁ 可输入、#6 时间表 UI、#7 失访率上浮等留待后续轮次。

### 改动
- `PlanModule.tsx`（#1）— 新增 `withSampleSize(base)` 助手：当用户已用「使用此参数」确定 N（`ssChosen>0`）时，把「每组 N、合计、设计场景、α、power、效应量」组织成一句事实追加进生成载荷的 `resources`，并提示临床按失访率上浮。`submit()`（生成实验计划）与 `genSap()`（生成 SAP）改为传 `resources: withSampleSize(resources)`。不改后端 schema（`resources` 本就透传进提示词），零 API 契约变更。
- `PlanModule.tsx`（#1 文案）— 「已保存到方案：N = X」→「✓ 已采用 N = X（每组）——生成『实验计划』/『SAP』时会带入此样本量」，承诺变为真实且说明它作用在哪。
- `PlanModule.tsx`（#5）— form-actions 下方新增门槛提示：`!idea.trim()` 时显示「开始前请先填写你的研究想法 / 课题，才能生成实验计划、SAP、DMP 或知情同意书」。
- `styles.css` `.form-actions`（#4a）— 加 `flex-wrap: wrap`：5 个按钮在窄窗/缩放下自动换行，不再横向溢出（此改为全局，grant 等复用 `.form-actions` 的模块同样受益）。

### 预期效果
- 用户在计算器里确定的样本量，真正进入 AI 生成的实验方案与 SAP，不再出现「方案里的 N 与我算的对不上」。
- 禁用按钮能自解释，首次使用不再对着灰按钮发懵。
- 生成按钮行在窄屏不溢出。
- `npm run build` 通过（tsc + vite，无报错）。

### 未做 / 留待后续轮次（plan 第 2 圈优先）
- **#3（高）**：把双比例的基线率 p₁ 从硬编码 0.3 改成可输入（对样本量计算影响最大）。
- **#2**：把样本量计算器上移到 AI 方案之前，或加「先算样本量→再生成」的引导，修复工作流倒置。
- **#7**：样本量结果加失访率上浮（10–20%）与分配比例调整。
- **#8/#9**：近似 N 加「近似值，点使用此参数做精确验证」标注；验证结论从小灰字提级。
- #6 结构化时间表 UI；#11 效应量滑块按场景重置/区分；#4b SAP/DMP 缩写就地解释；#12 报错友好化。

## 第 2 圈 · 2026-07-02

### 起因（批判要点，聚焦计算正确性 bug）
派 agent 二轮评审，重点核算样本量计算器的数值正确性。实测确认多个严重问题：

**严重·正确性**
1. **前端大字 N 与后端「精确」值系统性差 2×**：前端 Lehr（合并方差+率差），后端 arcsine/Cohen's h。默认场景（p₁=0.3,p₂=0.6,α=0.05,power=0.8）前端每组 44、后端每组 21。这不是近似误差，是效应量定义不同导致的 2× 系统差，却以 `fontSize:20` 大字当权威展示。
3. **【新】「使用此参数」后拖滑块不重新确认 → 注入方案的 N 与 α/power/效应量自相矛盾**：`withSampleSize` 用固化的 `ssChosen`，但 α/power/效应量取**实时滑块值**。用户在 effect=0.3 时固化 N=44，再拖到 0.6 不重点，就会注入「每组 44 例；效应量=0.6」这种内部打架的事实，LLM 照此论证，审稿一眼看穿。
5. **useThisN 先固化前端估值，后端精确值只用于「警告」从不被采用**，且警告塞进小灰字。系统算出了正确答案、显示了正确答案、却坚持用错误答案喂给 AI。
- **【reset 泄漏】** `reset()` 不清 `ssChosen`/场景/滑块，「清空」换新课题会把上一课题旧样本量悄悄注入下一份方案。

**其他（本轮未改，见下）**：#2 p₁ 硬编码 0.3、#4 效应量单滑块跨场景不重置/率差≥0.7 静默截断、#6 计算器在结果下方、#7 无失访率上浮、#8 扫描曲线轴范围写死。
（健壮性 OK：`useThisN`/`genRandomize` 都有 try/catch，不会锁死。）

### 本轮取舍
聚焦「**真正注入 AI 方案的那个样本量必须正确且自洽**」这一条主线，一次修掉 #1/#3/#5 + reset 泄漏（都围绕 ssChosen 这个注入值）；p₁ 可输入(#2)、失访率上浮(#7)、计算器上移(#6) 等留待后续。

### 改动
- `PlanModule.tsx` `useThisN`（#1/#5）— 点「使用此参数」时**采用后端精确值** `j.per_group` 作为 `ssChosen`（而非可能相差一倍的前端 Lehr 近似）；后端不可用才回退前端估算并标注来源。注入方案的数从此是本地精确计算值。
- `PlanModule.tsx`（#3）— 新增 `ssChosenMeta` 快照：确定 N 的同时固化当下的 α/power/效应量/场景/来源；`withSampleSize` 改用快照而非实时滑块值，消除「N 与参数对不上」的自相矛盾。
- `PlanModule.tsx` `reset()`— 一并清 `ssChosen`/`ssChosenMeta`/验证信息，杜绝旧样本量跨课题泄漏。
- `PlanModule.tsx` 结果显示（#1 诚实化）— 大字 `N =` 改为 `N ≈`，并明确标注「这是快速近似，点『使用此参数』会用本地精确计算得到并采用的 N」，避免把可能差一倍的估算当权威。

### 预期效果
- 真正写进实验方案/SAP 的样本量是本地精确计算值，不再可能比正确值大一倍。
- 注入的 N 与 α/power/效应量始终来自同一次确认，不再自相矛盾。
- 「清空」后换课题不会误带上一课题的样本量。
- 大字近似值不再被误当权威结论。
- `npm run build` 通过。

### 未做 / 留待后续轮次（plan 第 3 圈优先）
- **#2（高）**：双比例暴露 p₁ 输入框（前后端都还硬编码 0.3；本轮只做到前后端一致，未解决 p₁ 本身可能不对）。
- **#4**：切换场景重置效应量、双比例滑块上限随 (1−p₁) 收窄、截断给可见警告。
- **#7** 失访率上浮（确定性 `ceil(N/(1−dropout))`，大字并列名义/上浮后）；**#6** 计算器上移到生成按钮之前；**#8** 扫描曲线轴范围随场景。
