# 入口：回复审稿 (rebuttal)

拆解审稿意见 → 逐条生成 point-by-point 回复信（本地处理、数据不出网）。
代码：`frontend/src/modules/RebuttalModule.tsx`；后端：`backend/app/rebuttal.py`；样式：`frontend/src/styles.css`。

---

## 第 1 圈 · 2026-07-02

### 起因（批判要点）
派 agent 从「非 IT 医学/药学研究者第一次回复审稿人」视角批判，并与已打磨模块做一致性对比。先取证：后端「意见清单」与「回复信」是两条独立 LLM 调用，回复信拿 reviews 原文另起炉灶生成、并不消费结构化 comments，所以前端「逐条」其实是一整封 Markdown blob、与意见清单无编号绑定。12 条，摘录：

**高**
1. **结果区没有「复制」按钮**——与全站其他模块（idea/analyze/imrad/journal）不一致；回复审稿最高频动作就是复制回投稿系统。
2. **默认「用中文输出」，但 rebuttal 信绝大多数期刊要英文**（`rebuttal.py:91` 硬编码）；前端只有语气下拉、无语言选项。近乎致命默认值。
3. **报错只有红框、无「重试」**；审稿意见+稿件是长文本，流式易超时断流，失败后无法一键重发。
4. 拆条完全不可调（只读 `<ol>`），且与回复信条数对不上。
5. 无逐条编辑/逐条重生成，改一条得整封重来、还会触发 Diff 覆盖。

**中**
6. 必填按钮置灰但不解释原因。
7. 两个上传行为不一致（意见=追加、稿件=覆盖且无提示）+ 稿件框 rows=3 太小、与「全文」诉求矛盾。
8. 「清空」一键清光、无二次确认。
9. 意见清单与回复信分处两栏、无编号呼应，难逐条核对。

**低**
10. 占位/术语措辞不一致（「左侧」方向词错、稿件全文/原稿混用）。 11. 类型标签无解释、拆条阶段反馈粗（长意见解析期间只能盯 spinner）。

### 本轮取舍（第 1 圈收官，覆盖三项高频刚需 + 门槛提示）
挑三项高、且能复用站内既有模式的刚需：**#1 复制、#2 语言选项、#3 重试**，外加一致性的 **#6 门槛提示**。#4/#5（拆条可调、逐条编辑重生成）是较大的交互重构，留待后续轮次。

### 改动
- 后端 `rebuttal.py`（#2）— `_letter_messages` 增 `lang` 形参；把硬编码的「用中文输出。」改为按 `lang` 切换（`en` → "Write the entire response letter in English (fluent, formal academic English)."，否则中文）。`rebuttal()` 读 `inputs.get("lang")` 并传入。
- 前端 `RebuttalModule.tsx`（#2）— 新增「回复信语言」下拉（English / 中文），随请求 `{reviews, manuscript, tone, lang}` 传入（`streamRebuttal` 透传整个 inputs，无需改 sse.ts）。**默认改为 English**（多数期刊要求），中文一键可切。
- 前端（#1）— 结果工具栏加「复制整封信」按钮（带「已复制 ✓」反馈），并把工具栏按钮用 `.result-actions` 编组。
- 前端（#3）— 报错框加「重试」按钮（调 `submit`，输入仍在可一键重发）。
- 前端（#6）— 审稿意见为空时门槛提示「请先粘贴审稿意见（或上传文件）」。

### 预期效果
- 回复信可一键复制回投稿系统/邮件，补齐与全站一致的取出能力。
- 默认产出英文回复信，契合真实投稿场景；需要中文一键可切——修掉近乎致命的默认值。
- 流式失败可一键重试，不必重新面对长表单。
- 灰按钮点不动时知道缺什么。
- `npm run build` 通过；`rebuttal.py` 语法校验通过。

- **#7** 统一两个上传行为 + 稿件框加大；**#8** 清空二次确认；**#9** 意见清单与回复信编号呼应/锚点跳转；**#10/#11** 措辞统一 + 类型标签 tooltip + 拆条渐进反馈。
