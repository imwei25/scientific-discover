# 自动改进循环指令

把下面「指令正文」整段粘贴到 `/loop` 后面运行（不带时间间隔 = 让模型自己掌握节奏）：

```
/loop 按 改进日志/LOOP改进指令.md 的"指令正文"执行一轮自动改进
```

或直接把下面正文粘贴到 `/loop ` 之后。

---

## 指令正文

你是科研助手项目的自动改进循环。**每次被唤醒只完成一个改进方向**，保持小步快跑。严格按以下步骤：

### 1. 读取状态
- 读 `改进日志/BACKLOG.md`（待办与遗留问题）和 `改进日志/LOG.md`（已做过什么，避免重复）。
- 若 BACKLOG「任务拆解」为空 → 先做一次拆解：把"持续改进科研助手"拆成 4-6 个子任务（如：检索质量、生成质量、前端体验、性能、稳定性/错误处理、部署易用性），每个子任务先列 2-3 个具体改进方向写入 BACKLOG，然后本轮结束（拆解本身算一轮，commit+push 后停下等下一轮）。

### 2. 选一个方向
- 优先处理「遗留问题」里的 bug，其次按顺序取第一个 `[ ]` 方向。标记为 `[~]` 进行中。
- 若所有项都是 `[x]`/`[-]`，进入第 6 步「收尾判定」。

### 3. 调研
- 针对该方向，**先调研再动手**：读相关源码（backend/frontend/src-tauri），必要时用 WebSearch 查更好的做法/库/算法。
- 想清楚"改成什么样、为什么更好、怎么验证"，再改。

### 4. 改进
- 做**最小可用**的改动实现这一个方向。不顺手改无关代码。
- 保持与现有代码风格一致。

### 5. 像真实用户那样测试
- 不是只跑单测——要模拟真实用户路径验证：启动应用 / 调对应接口 / 走一遍前端操作，观察实际输出与体验。
- 测出问题：要么本轮修掉，要么作为新条目记进 BACKLOG「遗留问题」。
- 测试通过后：
  - 在 BACKLOG 把该方向标 `[x]`（放弃则 `[-]` 并写原因）。
  - 在 `改进日志/LOG.md` **置顶追加一条**日志（用文件里的模板：动机/改动/测试步骤与结果/commit）。
  - `git add -A && git commit` （信息：`改进(<子任务>): <方向标题>`）然后 `git push`。
- **一轮只到这里。** 本轮结束。

### 6. 收尾判定（仅当 BACKLOG 再无 `[ ]` 项时）
- 复盘：对每个子任务再想一遍"还有没有可改进方向？真实用户还会在哪卡壳？"
  - **有** → 追加新方向到 BACKLOG，继续循环。
  - **没有，且遗留问题清单为空** → 输出总结，**停止循环**（不再安排下一轮）。

### 硬性规则
- **Python 一律用项目 `.venv`**（位于 `backend/.venv`）：跑脚本/起后端用 `backend/.venv/Scripts/python.exe`，装包用 `backend/.venv/Scripts/pip.exe`。绝不用全局 python。
- **测试默认 mock 不花钱**：真实用户测试优先用 `MOCK_LLM=true` 或 `selftest.py mock` / Playwright，避免消耗 API 额度。
- 每轮必须：① 真实测试过 ② 写日志 ③ commit+push。三者缺一不算完成。
- 一轮一个方向，避免上下文膨胀。改不动/测不过就回退改动并把该方向记为 `[-]`，不要硬塞。
- 任何 push 失败或测试无法进行 → 停下来报告，不要假装成功。
```
